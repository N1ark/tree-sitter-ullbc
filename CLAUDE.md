# tree-sitter-ullbc — maintenance guide

Tree-sitter grammar for **Charon's** pretty-printed IR crate dumps (ULLBC and
LLBC dialects). It exists to syntax-highlight `charon/tests/ui/*.out` dumps, so
the grammar is deliberately *tolerant*: it aims to parse every real dump with
zero `(ERROR ...)`/`(MISSING ...)` nodes, not to be a faithful semantic model.

Its correctness is pinned by one thing: the **corpus CI** (`.github/workflows/
corpus.yml`) parses every IR dump in Charon's test suite and fails on any parse
error. Because Charon's pretty-printer evolves, this repo can go red **without
any change here** — a new syntactic construct lands in Charon and our grammar
doesn't know it yet. This guide is how you fix that.

## Layout

| Path | What it is |
|------|-----------|
| `grammar.js` | The grammar. **This is the only source file you edit.** |
| `src/parser.c`, `src/grammar.json`, `src/node-types.json`, `src/tree_sitter/` | Generated by `tree-sitter generate`. Committed (Zed compiles `src/parser.c`). Never hand-edit. |
| `queries/highlights.scm` | Highlight queries (node names must track grammar changes). |
| `scripts/corpus.sh` | The corpus validate/debug tool. Read its `--help`. |
| `scripts/tested-charon.txt` | The last Charon commit this grammar was validated against. Updated automatically on a clean full run. |
| `.github/workflows/corpus.yml` | CI: runs `scripts/corpus.sh --group` against Charon `main`. |

The grammar is consumed by the Zed extension **`ullbc-syntax`**, which pins a
commit of this repo in its `extension.toml` (`[grammars.ullbc] rev = ...`).
Shipping a grammar fix to Zed = push here, then bump that `rev` (see the last
section).

## The tool: `scripts/corpus.sh`

**Always start here. Do not hand-roll `find | grep | tree-sitter parse`
pipelines — the tool already does it, and better.** `scripts/corpus.sh --help`
is the source of truth; the essentials:

```sh
npm install                                  # once, to get the tree-sitter CLI

scripts/corpus.sh --charon ../charon         # full run against a local checkout
scripts/corpus.sh --charon ../charon --list  # just the failing file paths
scripts/corpus.sh --charon ../charon --group # histogram of failing *constructs*  <-- start debugging here
scripts/corpus.sh --charon ../charon simple/hello-world.out   # drill into one file (auto-detailed)
scripts/corpus.sh                            # CI form: clone Charon@main, check everything
```

`--group` is the key debugging mode: it collects the offending source line of
every parse error, normalises away local names/numbers/strings, and prints a
ranked histogram of *distinct* constructs with an example `FILE:LINE` each. One
look tells you "what new syntax do I need to support" instead of scrolling
hundreds of per-file errors.

The tool regenerates the parser from `grammar.js` on every run (pass
`--no-generate` to skip). It always prints the Charon commit under test, and
after a clean full run writes it to `scripts/tested-charon.txt`.

## Fix workflow (Charon added new syntax → CI red)

1. **Reproduce & localise.** With a local Charon checkout at `../charon`:
   ```sh
   scripts/corpus.sh --charon ../charon --group
   ```
   The histogram shows the failing construct(s) and example locations.

2. **See it in context.** Open an example `FILE:LINE`, and for the full parse
   tree of one file:
   ```sh
   scripts/corpus.sh --charon ../charon <that-file>       # error detail + carets
   ./node_modules/.bin/tree-sitter parse ../charon/charon/tests/ui/<file>  # full sexp
   ```

3. **Find how Charon prints it.** The pretty-printer lives at
   `../charon/charon/src/pretty/` (mainly `fmt_with_ctx.rs`). Grep it for the
   literal token or the keyword to learn the *exact* shape, delimiters, and —
   crucially for a non-indentation-sensitive grammar — what **grammatically
   bounds** the construct (a closing brace, a terminator keyword, etc.).
   > Example already handled: the `↳⚡ ` on-unwind block (`LLBC_UNWIND_PREFIX`
   > in `fmt_with_ctx.rs`). It has no closing delimiter, but an unwind handler
   > always ends in a diverging terminator (`unwind_continue` /
   > `unwind_terminate` / `undefined_behavior`), so `unwind_block` in
   > `grammar.js` is `seq('↳⚡', repeat($._statement), <diverging terminator>)`.
   > When a construct isn't brace-delimited, this "what ends it" question is the
   > whole game.

4. **Edit `grammar.js`.** Match the existing style (see the dialect notes in the
   file header). Prefer extending an existing rule to inventing a new top-level
   one. If `tree-sitter generate` reports an *unresolved conflict*, read its
   suggestion — usually a precedence tweak or a `conflicts:` entry. GLR + a
   declared conflict is fine here: the corpus only requires *a* valid,
   error-free parse.

5. **Regenerate & verify.**
   ```sh
   ./node_modules/.bin/tree-sitter generate     # must be conflict-free
   scripts/corpus.sh --charon ../charon          # must end "OK: all N ... parse cleanly"
   ```
   Don't trust a partial fix: confirm the full run is green (it also refreshes
   `scripts/tested-charon.txt`). Spot-check the parse tree of a couple of
   examples so the new nodes are structured sensibly, not just error-free.

6. **Keep highlights in sync.** If you added/renamed nodes that should be
   coloured, update `queries/highlights.scm`.

7. **Commit** `grammar.js`, the regenerated `src/*`, `scripts/tested-charon.txt`,
   and any query/CI changes together. Recording the Charon hash alongside the
   grammar makes future regressions bisectable: CI pins Charon `main`, so a red
   build means the break is somewhere between the recorded hash and current
   `main` — `git bisect` Charon across that range to find the pretty-printer
   change, then repeat from step 1.

## Gotchas

- **Not indentation-sensitive.** Tree-sitter ignores whitespace here (it's in
  `extras`). Never rely on indentation to bound a construct; find a real token /
  terminator boundary (see step 3).
- **Two dialects share most rules.** ULLBC statements end with `;` and use
  basic blocks + explicit terminators; LLBC is structured (`if`/`loop`/`match`)
  with no `;`. `semi()` is `optional(';')` so both parse. New statement forms
  usually need to work in both.
- **The external scanner** (`src/scanner.c`) supplies `name_group` (`{...}` name
  disambiguators). It's rarely the culprit, but if a `{...}` name fails to
  tokenise, look there.
- **`.out` files that aren't IR** (rustc panics/diagnostics) are skipped by the
  tool: only files whose first line is `# Final {U,}LLBC before serialization:`
  are checked. A "0 dumps found" error means you pointed it at a non-Charon dir.

## Shipping to the Zed extension

The `ullbc-syntax` extension fetches this repo at a pinned commit:

1. Push the grammar fix here (green CI on the pushed commit).
2. In `ullbc-syntax/extension.toml`, set `[grammars.ullbc] rev` to the new
   commit hash. Zed rebuilds the grammar wasm from that rev — you do **not**
   commit a `.wasm` from this repo (they're gitignored build artifacts).
